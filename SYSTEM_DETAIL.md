# 系统详细讲解

更新时间：2026-04-01

## 1. 系统目标

本项目用于数字孪生地铁灾害应急仿真，当前联调目标是完成以下闭环：

1. 前端发起仿真创建与播放控制。
2. 中台记录仿真与帧数据，并提供查询接口。
3. UE 通过 Pixel Streaming 推流，前端可见可控。

## 2. 三部分架构与职责

### 2.1 UE 客户端（画面生产者）

- 负责场景渲染与像素流输出。
- 通过 Pixel Streaming 参数连接信令服务的 Streamer 端口（8888）。
- 不直接面向业务管理页面，不负责数据库存储。

### 2.2 前端信令服务 + 页面（交互入口）

- SignallingWebServer 提供浏览器访问入口（8080）和 UE 推流接入（8888）。
- 前端页面负责：仿真配置输入、状态展示、历史数据浏览、分析配置交互。
- 前端同时与中台进行 HTTP/Socket 通信，以获取业务数据和实时状态。

### 2.3 中台服务（业务与持久化）

- 提供仿真管理 API、历史查询 API、实时推送能力。
- 负责 MySQL 持久化：simulations、simulation_frames、plans、plan_runs 等。
- 作为业务规则中心，承载数据约束、配置中心、流程编排。
- 监听端口由 `.env` 的 `PORT` 决定（当前仓库为 `3100`，代码默认值为 `3000`）。

## 3. 为什么同时使用 HTTP 和 Socket

你提出的疑问是合理的，当前建议保留“双通道”，职责拆分如下：

1. HTTP（请求-响应）
- 用于创建仿真、查询历史、按时间取帧、健康检查等确定性操作。
- 优点：可重试、可审计、易做权限和幂等。

2. Socket（事件推送）
- 用于实时播放状态、进度广播、异步事件通知（如仿真结束）。
- 优点：低延迟、服务端主动推送，适合回放和联动场景。

结论：
- 如果只留 HTTP，前端需要高频轮询，实时性和效率变差。
- 如果只留 Socket，查询型接口和审计链条会变得复杂。
- 当前阶段保留两者是工程上稳妥方案。

## 4. 当前实现状态（第二轮结束）

### 已完成

- UE、信令服务、中台三部分可连接。
- 前端新建仿真页面配置项较完整。
- 历史数据页已从弹窗拆分为独立页面并可展示时序数据。

### 未完成或仅占位

- 历史数据的数据分析和真实绘图能力尚未实现。
- 数据库闭环证据仍需按用例逐项补齐。
- 特殊实体固定位与可选项尚未由中台统一下发。

## 5. 建议的目标职责边界

针对“特殊实体位置编号谁来定”的问题，建议明确为：

1. UE 场景设计者
- 提供场景内合法点位、实体类型、容量约束的原始数据。

2. 中台
- 维护统一配置（实体模板、可放置点位、版本号）。
- 对外暴露配置查询接口并负责校验前端提交。

3. 前端
- 初始化时拉取配置并渲染交互，不硬编码业务规则。
- 仅在 UI 层做可用性限制，不替代中台校验。

## 6. 下一阶段最小闭环建议

- 先实现“模拟仿真后端”作为帧数据来源（可不含真实仿真算法）。
- 中台负责创建仿真、接收并落库存储模拟帧、广播结束事件。
- 前端通过历史页和回放链路验证数据可见性。
