# 系统详细讲解

更新时间：2026-04-17

## 1. 系统目标

本项目用于数字孪生地铁灾害应急仿真，当前联调目标是完成以下闭环：

1. 前端发起仿真创建与播放控制。
2. 中台记录仿真与帧数据，并提供查询接口。
3. UE 通过 Pixel Streaming 推流，前端可见可控。
4. 模拟后端与模拟LLM可替代真实算法链路完成联调验证。

## 2. 三部分架构与职责

### 2.1 UE 客户端（画面生产者）

- 负责场景渲染与像素流输出。
- 通过 Pixel Streaming 参数连接信令服务的 Streamer 端口（8888）。
- 不直接面向业务管理页面，不负责数据库存储。

### 2.2 前端信令服务 + 页面（交互入口）

- SignallingWebServer 提供浏览器访问入口（8080）和 UE 推流接入（8888）。
- 前端页面负责：仿真配置输入、状态展示、历史数据浏览、分析配置交互。
- 前端同时与中台进行 HTTP/Socket 通信，以获取业务数据和实时状态。

### 2.3 中台服务（业务与持久化）

- 提供仿真管理 API、历史查询 API、实时推送能力。
- 负责 MySQL 持久化：simulations、simulation_frames、plans、plan_runs 等。
- 作为业务规则中心，承载数据约束、配置中心、流程编排。
- 监听端口由 `.env` 的 `PORT` 决定（当前仓库为 `3100`，代码默认值为 `3000`）。

### 2.4 模拟服务（过渡期联调支撑）

- Mock Backend（3200）接收中台触发，按固定节奏回写 `FrameSnapshot`。
- Mock LLM（3300）接收中台请求，返回合法 `PlanConfig` 并可在中台落库。
- 该层仅用于测试阶段，不替代真实仿真与真实大模型实现。

## 3. 为什么同时使用 HTTP 和 Socket

你提出的疑问是合理的，当前建议保留“双通道”，职责拆分如下：

1. HTTP（请求-响应）
- 用于创建仿真、查询历史、按时间取帧、健康检查等确定性操作。
- 优点：可重试、可审计、易做权限和幂等。

2. Socket（事件推送）
- 用于实时播放状态、进度广播、异步事件通知（如仿真结束）。
- 优点：低延迟、服务端主动推送，适合回放和联动场景。

结论：
- 如果只留 HTTP，前端需要高频轮询，实时性和效率变差。
- 如果只留 Socket，查询型接口和审计链条会变得复杂。
- 当前阶段保留两者是工程上稳妥方案。

## 4. 当前实现状态（第二轮结束 + 模拟服务接入）

### 已完成

- UE、信令服务、中台三部分可连接。
- 中台与 Mock Backend/Mock LLM 已打通集成接口。
- 前端已提供“启动模拟仿真”触发入口，并接入中台 LLM 集成流程。
- 前端新建仿真页面配置项较完整。
- 历史数据页已从弹窗拆分为独立页面并可展示时序数据。
- 仿真最后一帧完成态已可回传，前端可显示“仿真计算完成”提示。

### 未完成或仅占位

- 历史数据的数据分析和真实绘图能力尚未实现。
- 数据库闭环证据已形成首轮验证样例，尚需扩大覆盖面（高帧数、异常回写、权限场景）。
- 特殊实体固定位与可选项尚未由中台统一下发。

## 5. 建议的目标职责边界

针对“特殊实体位置编号谁来定”的问题，建议明确为：

1. UE 场景设计者
- 提供场景内合法点位、实体类型、容量约束的原始数据。

2. 中台
- 维护统一配置（实体模板、可放置点位、版本号）。
- 对外暴露配置查询接口并负责校验前端提交。

3. 前端
- 初始化时拉取配置并渲染交互，不硬编码业务规则。
- 仅在 UI 层做可用性限制，不替代中台校验。

## 8. 下一阶段详细设计说明（跨部门协作口径）

### 8.1 真实后端接入原则

1. 真实后端替换 Mock Backend 时，保持中台上游协议稳定，优先兼容：
- `POST /api/integration/backend/start`（中台入口）
- 中台下游批量写帧接口（由中台内部或服务间协议承接）
2. 数据对象以 `FrameSnapshot` 为唯一落库标准对象，禁止新增“仅前端可读”的临时字段。
3. 若真实后端提供增量帧，中台需在落库前还原/补齐可回放快照能力。

### 8.2 真实大模型接入原则

1. 真实大模型替换 Mock LLM 时，保持中台入口稳定：
- `POST /api/integration/llm/plan`
2. 大模型返回必须通过中台 schema 校验后入库，非法字段按审计日志记录并拒绝 `apply`。
3. 预案必须可追溯：记录请求参数、模型版本、响应摘要、失败原因。

### 8.3 历史可视化实现口径

1. 历史页图表数据来源：
- `GET /api/simulations/:simId/info`
- `GET /api/simulations/:simId/frame?time=...`
2. 图表建议首批实现：
- 疏散趋势：`statistics.totalEvacuated` vs `simTime`
- 风险趋势：`statistics.casualtyCount` / `environment.zones[].gasConcentration`
- 区域热力：`agents[].pos` 聚合
3. 数据量控制：默认采样，避免一次性拉取全量帧导致页面卡顿。

### 8.4 交付与验收建议

1. 每个子系统交付需附：接口说明、字段样例、失败码说明。
2. 联调验收需附：启动日志、端口检测、关键 API 响应、SQL 查询截图或导出。
3. 回归基线统一以 `VISUAL_TEST_RUNBOOK.md` 与 `DOCS/PROCESS/NEXT_PHASE_EXECUTION_PLAN.md` 为准。
