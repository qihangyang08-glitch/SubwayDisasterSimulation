## 模块一：数据库持久化与模型层（Node.js + MySQL）
1) 目标
- 持久化 InitConfig、FrameSnapshot、PlanConfig 原始 JSON。
- 支持按时间与按帧索引检索（回放与跳转）。
- 提供 plan/plan_run 关系查询能力。
- 时间语义对齐 DataStruction：`createdAt` 对应毫秒时间戳（DB 列 `created_at` 使用 BIGINT）。
1.1) 交付物（用于提示词）
- 数据库表结构与索引设计（DDL 级别）。
- Repository/DAL 接口列表与入参出参定义。
- 幂等写入与事务边界说明。
1.2) 任务清单（顺序）
- 定义表结构与索引（含 JSON 字段与联合主键）。
- 明确字段映射规则：DB snake_case -> 上层 camelCase。
- 连接池与事务封装（plan + plan_run 同事务）。
- 实现 Repository 接口：createSimulation、batchInsertFrames、createPlan、createPlanRun。
- 实现查询接口：getFrameByTime、getFrameByIndex、getSimulationTimeRange、getPlanRuns、listSimulations。
- 幂等写入策略：批量插入冲突处理（忽略或幂等更新）。
- 最小验收用例与失败回滚测试。
2) 输入契约（字段与示例）
- createSimulation(initConfig: InitConfig)
- batchInsertFrames(simulationId, frames[])，frames[i] = FrameSnapshot
- createPlan(planConfig: PlanConfig)
- createPlanRun({baseSimulationId, planId, newSimulationId})
3) 输出契约（字段与示例）
- getFrameByTime(simulationId, targetTime) -> FrameSnapshot
- getFrameByIndex(simulationId, frameIndex) -> FrameSnapshot
- getSimulationTimeRange(simulationId) -> {minSimTime, maxSimTime}
- getPlanRuns(baseSimulationId) -> planRuns 列表（按 createdAt 倒序）
- listSimulations() -> [{simulationId, scenarioId, mapLevel, status, createdAt, minSimTime?, maxSimTime?}]
4) 依赖契约（只能依赖哪些模块）
- 仅提供 DAL/Repository 能力，不暴露 HTTP/Socket。
- 表列可用 snake_case，但对上层返回对象必须是 camelCase。
5) 非功能约束（性能/幂等/重试/兼容）
- 主键：simulation_frames(sim_id, frame_index)。
- 二级索引：simulation_frames(sim_id, sim_time)、plans(from_simulation_id)。
- 支持批量写入、连接池、超时、重试、事务（plan + plan_run 场景）。
- batchInsertFrames 需保证幂等（相同主键重复写入不污染）。
- 数据对象字段遵循 DataStruction camelCase 定义；DB 内部统一使用 snake_case，并通过 mapper 转换后再对上层返回 camelCase。
- 建议表：
  1) simulations(sim_id PK, scenario_id, map_level, status, init_config JSON, created_at BIGINT, finished_at BIGINT NULL)
  2) simulation_frames(sim_id, frame_index PK part, sim_time, frame_snapshot JSON, created_at BIGINT)
  3) plans(plan_id PK, from_simulation_id, from_sim_time, plan_source, plan_config JSON, created_at BIGINT)
  4) plan_runs(plan_run_id PK, base_simulation_id, plan_id, new_simulation_id, created_at BIGINT)
- 命名冲突消解：DataStruction 中的 `simulationFrames/simulationId/frameSnapshot/createdAt` 为上层对象命名；数据库落地分别对应 `simulation_frames/sim_id/frame_snapshot/created_at`。
6) 最小验收用例（3 条）
1. createSimulation 后时间范围为空。
2. batchInsertFrames 后，getFrameByTime 命中最近历史帧。
3. createPlan + createPlanRun 后可按 baseSimulationId 查到关联。
7) 失败场景与回滚策略
- 批量写入中断：整体事务回滚并可重试。
- 连接池不可用：降级为重试队列或返回错误给上层。
- 重复帧写入：忽略或幂等更新，不影响检索。

## 模块二：HTTP API 协同层（Express/Koa）
1) 目标
- 承接前端控制与仿真注入请求，统一校验、鉴权、错误码与 Envelope。
- 对外仅提供 HTTP，不直接操作 UE。
1.1) 交付物（用于提示词）
- HTTP 路由表（方法、路径、请求体、响应 Envelope）。
- 校验规则清单与错误码表。
- 兼容路由映射表与弃用策略。
1.2) 任务清单（顺序）
- 统一 Envelope 生成与错误封装（含 requestId 与 sentAt）。
- 路由主版本实现（/api/simulations 与 /api/plans）。
- 参数校验实现（ajv/Joi）并覆盖最小校验规则。
- 鉴权与 CORS（可开关）。
- 兼容路由映射与弃用提示。
- 幂等键处理（基于 requestId）。
- 最小验收用例与错误场景覆盖。
2) 输入契约（字段与示例）
- POST /api/simulations               body=InitConfig
- POST /api/simulations/:simId/frames body={frames:[FrameSnapshot]}
- GET  /api/simulations               查询历史仿真列表
- GET  /api/simulations/:simId/info
- GET  /api/simulations/:simId/frame?time=12.5
- GET  /api/simulations/:simId/frame/:frameIndex
- POST /api/plans                     body=PlanConfig
- POST /api/plans/:planId/apply       body={fromSimTime?}
- 向后兼容别名（仅过渡期）：
  - POST /api/init -> POST /api/simulations
  - GET /api/sim/:simulationId/state -> GET /api/simulations/:simId/info
  - GET /api/frame/:simulationId/:frameIndex -> GET /api/simulations/:simId/frame/:frameIndex
  - POST /api/plan -> POST /api/plans
  - POST /api/plan/apply -> POST /api/plans/:planId/apply
3) 输出契约（字段与示例）
- 所有响应统一 Envelope：{version, requestId, simulationId, messageType, sentAt, payload}
- messageType 约定：
  - /frame?time 或 /frame/:frameIndex -> FrameSnapshot
  - /info -> Ack（payload={minSimTime,maxSimTime,status?}）
  - /simulations (GET) -> Ack（payload={simulations:[...] }）
  - /simulations -> Ack（payload={simulationId}）
  - /plans -> Ack（payload={planId}）
  - /plans/:planId/apply -> Ack（payload={planRunId,newSimulationId?}）
- simulationId 规则：与仿真相关时必填；非仿真接口使用空字符串。
4) 依赖契约（只能依赖哪些模块）
- 仅调用模块一 DAL/Repository。
5) 非功能约束（性能/幂等/重试/兼容）
- Joi/ajv 校验，统一错误码与错误 Envelope（messageType=Error）。
- CORS、可选 Token 鉴权、请求幂等键（建议基于 requestId）。
- 所有字段命名遵循 DataStruction camelCase。
6) 最小验收用例（3 条）
1. 创建 simulation + 注入 frames 后，/info 返回 maxSimTime。
2. /frame?time=t 返回最近不超过 t 的帧。
3. create plan + apply 返回 Ack 型占位确认。
7) 失败场景与回滚策略
- 参数校验失败：返回 400 + Error Envelope。
- simulationId/planId 不存在：返回 404 + Error Envelope。
- DAL 写入失败：返回 500 + Error Envelope，并记录重试队列。

## 模块三：中台高频调度器（Socket.IO Dispatcher，1~2s/帧）
1) 目标
- 维护播放状态机，向前端与 UE 广播帧与状态。
- 转发前端控制指令至 UE（仅中台链路）。
1.1) 交付物（用于提示词）
- 播放状态机定义（状态、事件、转移）。
- Socket 事件契约表（事件名、Envelope.messageType、payload 结构）。
- 时间推进与乱序处理规则说明。
1.2) 任务清单（顺序）
- Socket.IO Server 初始化与命名空间/房间设计（按 simulationId）。
- 播放状态机实现（play/pause/seek/setSpeed）。
- Tick 调度器与时间推进公式落地。
- 从模块一读取帧与时间范围（getFrameByTime/getSimulationTimeRange）。
- UpdateFrame/SimState/Ack 事件发送与乱序过滤。
- 断线重连与补发当前帧逻辑。
- UE4.26 兼容性联调与冒烟回归。
2) 输入契约（字段与示例）
- play（Envelope.messageType=ControlCommand，payload={action:"play"}）
- pause（Envelope.messageType=ControlCommand，payload={action:"pause"}）
- seek{targetTime}（payload={action:"seek",targetTime}）
- setSpeed{speed}（payload={action:"setSpeed",speed}）
- ControlCamera{cameraCommand}（Socket 事件名=ControlCamera，Envelope.messageType=ControlCommand）
3) 输出契约（字段与示例）
- 所有事件统一 Envelope。
- Socket 事件名集合：UpdateFrame, SimState, ControlCamera, PlanCommand, Ack。
- Envelope.messageType 映射：
  - UpdateFrame -> FrameSnapshot
  - SimState -> SimState
  - ControlCamera -> ControlCommand
  - PlanCommand -> PlanCommand
  - Ack -> Ack
- UpdateFrame.payload = FrameSnapshot（不再套一层 frameSnapshot 字段）。
- SimState.payload = {state, currentTime, playbackSpeed}。
4) 依赖契约（只能依赖哪些模块）
- 仅读模块一 getFrameByTime/getSimulationTimeRange，不直连数据库以外资源。
5) 非功能约束（性能/幂等/重试/兼容）
- 时间推进：t_next = t_current + deltaT * playbackSpeed。
- 乱序防护：按 simTime 或 frameIndex 丢弃旧帧。
- 断线重连：新连接成功后立即补发当前帧 + SimState。
- UE 兼容：Socket.IO 协议与 UE4.26 客户端插件实测可连通版本一致。
6) 最小验收用例（3 条）
1. 2x 倍速 + 1s tick 时，仿真时间推进约 2s。
2. seek 后 1 个 tick 内收到目标附近帧。
3. pause 后 currentTime 不再增长。
7) 失败场景与回滚策略
- 帧缺失：回退到最近可用帧并发送 Ack 警告。
- 播放状态异常：重置为 pause 并广播 SimState。
- UE 断线：缓存最新帧，重连后补发。

6) 最小验收用例（3 条）
1. InitConfig 中任一特殊实体缺失 `triggerAt` 时返回 400。
2. PlanConfig 未提供 disasters 且基线存在时，apply 成功并返回 `inheritanceUsed=true`。
3. 同一动作重复下发时调度器仅执行一次并返回 Ack。
7) 失败场景与回滚策略
- 基线仿真不存在：返回 404 + Error Envelope，不写入 plan_run。
- 动作编译失败：返回 400/422，保留原始 plan 入库记录并标记 `compile_failed`。
- 调度执行失败：动作入重试队列，Ack 返回 pending。

## 模块四：前端客户服务程序（Web UI）
1) 目标
- 实现时间轴控制、数据可视化、键鼠采集、Pixel Streaming 播放。
- 支持两类策略编排入口：
  - 手工编排（直接编辑 InitConfig.specialEntities[].triggerAt 与动作参数）。
  - LLM 预案（提交上下文后生成 PlanConfig，由系统写入触发时刻与动作）。
1.1) 前端页面详细设计（正式版）
- 信息架构：采用“左导航 + 顶栏状态 + 主工作区 + 右侧详情抽屉”四区布局。
- 顶层导航入口调整为“左侧可收起侧边栏”统一承载，保留三类业务页面语义：
  - 初始配置（InitConfig Studio）
  - 历史查看（Replay & Inspect）
  - 大模型预案生成（LLM Plan Workbench）
- 顶栏统一元素：`simulationId`、连接状态灯（HTTP/Socket/Pixel Streaming）、当前 `simTime`、全局操作（保存草稿/发布）。
- 状态提示规则：
  - 成功提示使用顶部细条 + 右下角轻提示（2.5s 自动消失）。
  - 错误提示保留可展开详情（展示 `requestId` 与 Error Envelope.code）。

1.1.0) 主页面壳层设计优化（本轮新增）
- 页面主结构：
  - 左侧为可收起侧边栏（展开态建议 220~240px，收起态 56~64px，仅保留图标）。
  - 右侧为 UE 画面主视区，底部为视频式时间控制条。
- 侧边栏按钮（自上而下固定顺序）：
  - `+ 创建仿真`：打开悬浮子页面（Modal/Drawer）用于 InitConfig 编辑与校验。
  - `历史数据`：打开独立页面（新路由，可同窗口跳转或新标签）。
  - `大模型预案生成`：打开悬浮子页面用于 PlanConfig 设计与应用。
  - `快捷数据分析`：切换右侧视图区右上角“半透明分析仪表层”。
- 右侧主视区：
  - 上部固定为 Pixel Streaming 画面容器（保持现有 `#player` 节点）。
  - 下部为时间控制条：`后退`、`播放/暂停`、`前进`、`可拖动进度条`、当前值显示。
- 进度条单位策略（本轮选型）：
  - 主单位采用“帧步进（frameIndex）”，前进/后退默认步长为 1 帧（可切换 5/10 帧）。
  - 辅助显示采用“百分比 + simTime”，用于用户感知整体进度。
  - 选择依据：帧单位更利于与 `getFrameByIndex` 和 `UpdateFrame.frameIndex` 对齐，降低 seek 同步歧义。
- 快捷数据分析面板（新增约束）：
  - 位置：右上角悬浮，半透明，不遮挡核心控制条。
  - 数据来源：读取当前进度位置对应的 `FrameSnapshot`。
  - 默认展示字段：总人数、水位高度、烟雾范围、受影响人群、受影响程度。
  - 刷新节奏：跟随进度条变化与 `UpdateFrame` 推送，去抖建议 100ms。

1.1.1) 子页面一：初始配置（InitConfig Studio）
- 布局结构：
  - 左侧 25%：场景参数区（基础信息、人数、随机种子）。
  - 中间 50%：灾害配置区（water/fire 分栏，支持多 inlet/source 列表编辑）。
  - 右侧 25%：特殊实体编排区（按 `triggerAt` 升序时间线展示）。
- 核心控件：
  - 数值输入：`totalPeople`、`inflowRate`、`spreadSpeed` 使用步进输入框（支持键盘上下微调）。
  - 列表编辑：`water.inlets[]`、`fire.sources[]` 使用“表格 + 侧滑编辑面板”。
  - 特殊实体：每条卡片必须显示 `entityId`、`entityType`、`triggerAt`、关键参数摘要。
  - 时间线控件：支持拖动排序、批量平移触发时刻（+5s/+10s）。
- 按钮设计：
  - 主按钮：`创建仿真`（深蓝底、白字、无大圆角，右侧带提交状态点）。
  - 次按钮：`校验配置`、`保存草稿`、`导入JSON`、`导出JSON`。
  - 危险按钮：`重置当前页`（仅描边，需二次确认）。

1.1.2) 子页面二：历史查看（Replay & Inspect）
- 布局结构：
  - 上区 70%：左侧 Pixel Streaming 画面，右侧关键指标卡（疏散人数、伤亡、平均暴露时长）。
  - 下区 30%：时间轴 + 事件轨道 + ECharts 联动区。
- 时间控制条：
  - 控件顺序：`播放/暂停`、`单帧步进`、`倍速`、`时间输入框`、`拖拽轨道`。
  - 拖拽时每 100ms 节流触发 seek，请求结束后再触发下一次。

## 模块五：方案配置与接力（可选）
1) 目标
- 管理 PlanConfig 与 PlanRun，并触发接力仿真。
- 在“持久化/接力”和“动作下发”之间做解耦：先完成 apply（入库+接力），再执行 PlanCommand 投递。
- 明确预案语义：预案帧由 LLM 生成，核心是“特殊实体在何时执行何动作”；手工编排可通过 InitConfig 完成。
1.1) 交付物（用于提示词）
- PlanConfig/PlanRun 的入库与查询流程说明。
- apply 流程与 PlanCommand 事件映射表（两阶段：apply -> dispatch）。
- Ack/错误返回格式约定。
1.2) 任务清单（顺序）
- PlanConfig 校验与入库。
- apply 流程实现（仅负责记录 plan_run + 创建 newSimulationId，不与动作投递强耦合）。
- dispatch 流程实现（基于 planId/newSimulationId 触发 PlanCommand，支持重试与幂等）。
- Ack 返回与幂等处理（按 requestId）。
- 失败回滚与重试队列策略。
- 支持灾害配置继承：当 `initConfigLike.disasters` 缺失时从 `fromSimulationId` 查询并补齐。
- 预案动作编译：优先使用 `planRuntime.actions`，无 actions 时从 `specialEntities[].triggerAt` 生成动作队列。
2) 输入契约（字段与示例）
- PlanConfig 关键字段：fromSimulationId, fromSimTime, planSource, objective, initConfigLike, planRuntime.actions, expectedMetrics（`planId` 可由服务端生成或客户端传入）。
- `initConfigLike.disasters` 可选；`initConfigLike.specialEntities[].triggerAt` 必填且 >=0。
- apply 输入：{fromSimTime?}，planId 从路径读取。
- dispatch 输入（新增约束）：{planId, simulationId/newSimulationId, requestId}（可复用 Socket `PlanCommand` 事件）。
3) 输出契约（字段与示例）
- 写入 plans、plan_runs。
- apply Ack 返回接力结果：{planRunId, newSimulationId, fromSimTime, compiledActionCount?}。
- dispatch Ack 返回投递结果：{status, compiledActionCount, acceptedActionCount, dedupedActionCount}。
4) 依赖契约（只能依赖哪些模块）
- 调用模块一 plans/plan_runs；通过模块三转发 PlanCommand。
5) 非功能约束（性能/幂等/重试/兼容）
- apply 与 dispatch 分离幂等：两者均按 requestId 去重。
- Ack 幂等；与播放链路解耦；失败可重放（按 requestId）。
- PlanConfig 字段遵循 DataStruction 定义。
- 时序约束：planRuntime.actions.startAt 应不早于 fromSimTime。
- 时序约束补充：specialEntities.triggerAt 应不早于 fromSimTime。
- 语义约束：LLM 预案帧与手工 InitConfig 编排共存，但来源标识必须可追踪（planSource/manual 标记）。
6) 最小验收用例（3 条）
1. 提交 PlanConfig 成功入库。
2. apply 成功记录 plan_run。
3. dispatch PlanCommand 成功并收到 Ack（含 compiledActionCount）。
7) 失败场景与回滚策略
- planId 不存在：返回 Error Envelope 并记录失败原因。
- plan_run 写入失败：apply 阶段回滚事务；若已创建 newSimulationId，则标记为 `orphan_pending_cleanup` 并加入补偿队列。
- PlanCommand 投递失败：进入重试队列，Ack 返回 pending 状态；不回滚已完成的 apply。

## 全局统一约束（跨模块必须遵守）
- Envelope 固定字段：version, requestId, simulationId, messageType, sentAt, payload。
- Socket 事件集固定：UpdateFrame, SimState, ControlCamera, PlanCommand, Ack。
- messageType 与 DataStruction 对齐：InitConfig, FrameSnapshot, PlanConfig, ControlCommand, SimState, Ack, Error（可扩展 PlanCommand）。
- 数据校验：
  - fire.enabled=false 时，fire.sources 必须为空；
  - 概率值范围 [0,1]；速度/流量非负；
  - 同帧 agentId/entityId 唯一；
  - ext 保留但不得承载核心必填字段。
- 时间推进公式固定：t_next = t_current + deltaT * playbackSpeed。
- 摄像机控制唯一链路：前端 -> 中台 Socket.IO -> UE。
- UE4.26 稳定优先：版本冻结 + 兼容矩阵 + 冒烟回归。
